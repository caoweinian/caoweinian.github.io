Unsafe Rust的黑暗艺术

万般渴望的安全，最终匿于无形；黑暗与恐惧在我认为脚下的土地无比舒适的那最惬意的一刻轰然而降。狂风掠过，我睁眼看到血样的绘图：“安全记忆”的告示牌被撕碎；尖针飞出，刺向后方驰来的轮胎；猛踩住刹车，却只让仪表盘针向右飞转；任何键的第二次按下都不起任何作用；车随导航驶向无人循迹的深渊。原来承诺永无事故的轿车本身就在事故漩涡的中心。它正发出刺鼻的铜臭，水箱倾泻着带活物的酸液。我看着车窗右上角的“检验合格”，呆在原地，什么都做不了。

Rust死灵书深入挖掘所有你需要编写Unsafe Rust程序时需要了解的脏乱差细枝末节。

如果你还想要快乐地编写Rust程序，请现在马上回头，忘记这本书的存在——这本书实在不是你所需要的。然而，然而，如果你确实想写unsafe代码——或者仅仅想要挖掘Rust的本质——这本书有很多对你有用的信息。

与 Rust程序设计语言 不同，我们假定你有这些先行知识。特别是，你应该熟悉基本的系统编程以及Rust本身。如果你不熟悉这些主题，你应该先读 The Book 。不过，我们不假定你读过《The Book》 。我们会注意在适当的时候偶尔复习一下这本书涉及的基础知识。你大可以直接跳到这本书来读；不过有一点，我们不会再从头解释一切了。

遇见Safe与Unsafe

不用关心底层实现细节可真太幸福了。谁会关心一个空元组占用多少内存空间呢？很不幸，有时它真的很重要，这时我们不得不去了解它。如果你开始关心实现细节了，最常见的原因就是你希望程序性能得到提升。不过更重要的是，当直面硬件、操作系统或其他编程语言这些底层的东西时，这些细节可能影响程序的正确性！

当实现细节在一个safe的编程语言里变得很重要时，程序员通常有三条路可选：

  • 修改你的代码以使编译期/runtime更好地执行代码优化策略
  • 硬着头皮上，去采用更加单一或更加繁琐的设计来实现
  • 用另一种能够处理这些繁琐细节的编程语言重新实现你想要的一切

对于最后那一个选项，程序员会倾向于使用C语言。对于只对外声明了C语言函数接口的系统，程序员与该系统交互也几乎必须使用C语言。

不幸的是，使用C语言是非常非常非常危险的（不否认，有时你使用它是有充分理由的），这种不安在你想要和另一个语言进行互操作的时候会更加被放大。你必须小心仔细，去保证C语言和其他语言遵循了什么样的调用约定，而且他们之间不会互相使绊子埋雷。

这和Rust有什么关系？

怎么说呢，与C语言不同，Rust是一门安全（safe）的编程语言。

不过，与C语言类似，Rust也是一门不安全（unsafe）的编程语言。

更准确一点，Rust 包含 了安全（safe）与不安全（unsafe）编程语言。

Rust可以被认为是 Safe Rust 和 Unsafe Rust 两者的结合体。顾名思义：Safe Rust是安全的，Unsafe Rust是，唔，不安全的。事实上，Unsafe Rust允许我们做一些 真心 很不安全的事请。Rust的创造者们再三告诉你山有猛虎，奈何这本书的内容本身就是教你如何向山而行。

Safe Rust是 真正的 Rust语言。如果你只使用了Safe Rust部分，你就永远不用担心类型安全（type-safety）或内存安全（memory-safety）问题。你也完全不用担心悬垂指针（dangling pointer）、释放后使用（use-after-free），或者任何其他UB。

标准库给了你很多足够的现成工具让你能以最习惯方式通过Safe Rust编写高性能应用和库。

但你可能想要这个语言的另一面。也许你在写一个不暴露给标准库的低级抽象。也许你在写标准库本身（它全部由Rust编写）。也许你想要做一些类型系统（type-system）不理解的事，弄些古灵精怪的东西。也许……你需要Unsafe Rust。

Unsafe Rust和Safe Rust共享同样的语法规则和语义规则。而前者只是能让你多做一些 其他 事。这些事 非 常 危 险 （我们会在下一小节中定义这些概念）。

把两者分开的价值在于，我们受益于使用像C语言这样虽不安全却又能获得实现细节上的底层次控制，但又避免了由再使用另一门完全不同的安全编程语言而滋生的所有问题。

还有一些问题——最明显的是，我们必须留意类型系统在属性（properties）中作出的所有假设，并且在与Unsafe Rust交互的时候时刻记着它们。这就是本书的目的：告诉你这些假设是什么，以及怎么处理它们。

Safe与Unsafe如何交互

Safe Rust和Unsafe Rust之间有什么关系？它们是怎么交互的？

Safe Rust和Unsafe Rust之间的边界是通过 unsafe 关键字明确界定的，就像接口一样，是一方通向另一方的唯一通道。因此我们可以说Safe Rust是安全的编程语言：所有不安全的部分都在 unsafe 边界以后。如果你想，你可以在你的代码库中加入 #![forbid(unsafe_code)] 标注以静态方式（在编译期）保证你一直在写Safe Rust。

unsafe 关键字有两个用途：声明编译器无法检查的约定，以及郑重宣告——我，程序员，保证这些约定都被遵守了。

你可以通过使用 unsafe 来表明 函数 和 trait声明 存在未被编译器检查的约定。对于函数， unsafe 意味着使用本函数的用户必须仔细阅读该函数的文档，并保证用户严格遵守了此函数要求的约定。对于trait声明， unsafe 意味着实现此trait的对象必须仔细阅读该trait的文档，并保证该对象保证不违反此trait要求的约定。

你可以使用 unsafe 块来宣告，该块内所有的不安全（unsafe）的操作都被验证，确保符合这些操作的约定。比如，传给 slice::get_unchecked 的索引不能越界。

你可以使给trait实现标记为 unsafe 以宣示该实现遵守了trait的约定。举例来说，实现了 Send 的类型变量确实可以安全地被移动（move）到其他线程里。

标准库有大量的unsafe函数，包括：

  • slice::get_unchecked, 它在索引时不进行越界检查，这么做就可能导致内存安全（memory-safety）被轻易违反。
  • mem::transmute 将一个值解释为另一个给定的类型，从而以任意方式轻松绕过类型安全（type safety）（细节见 conversions）。
  • 所有指向具有确定大小结构（译者注：此“结构”是广义的，不仅指struct）的指针都有一个 offset 方法，如果你传递给该函数的偏移（offset）越界，则会引发UB。
  • 所有的FFI（Foreign Function Interface，外部函数接口）函数都是 unsafe 的，因为其他的编程语言所做的事到底安不安全，Rust没法检查。

Rust发布1.29.2版本时，标准库有定义下列unsafe traits（当然不仅限于下面这几个，不过其他的都尚未进入stable分支，而且其中有一些可能永远不会进入stable）：

  • Send 是一个没有API的标记trait（marker trait），并承诺所有实现它的结构（广义）可以被安全地在线程之间移动（move）。
  • Sync 是一个标记trait（marker trait），并承诺对于实现该trait的结构（广义），多个线程可以安全地共享该结构的引用（reference）。
  • GlobalAlloc 允许定制整个程序的内存分配器。

Rust标准库里有很多地方内部使用了Unsafe Rust。这些实现经过了严格的人工检查，因此基于这些不安全实现的Safe Rust接口可以被认为是安全的。

所有这些分离的需求都最终归结为一个Safe Rust的基本性质，即 正确性：

无论如何，Safe Rust永远不会产生UB。

Rust中安全与不安全互相分割的设计，其中暗含着它们之间存在“不对称信任”：Safe Rust必须相信它所直接接触的Unsafe Rust被正确地编写，而另一方面，Unsafe Rust在信任Safe Rust这一方面却总是如履薄冰。

举个例子，Rust有 PartialOrd 和 Ord traits，它们来区别那些能部分被比较的类型，以及那些提供了全序（total）的类型，后者意味着其比较表现是很合理的。

BTreeMap 对于只提供了偏序的类型没有意义，它要求比较键（key）实现了 Ord （译者注：即全序）。然而， BTreeMap 的内部实现有Unsafe Rust代码。因为我们无法接受让一个错误的 Ord 实现引发UB——毕竟我们只是在写Safe Rust（译者注：抬头再看一眼上面的粗体字！并且注意， Ord 是Safe trait）——那么BTreeMap里的Unsafe代码必须要非常健壮（robust），以使它针对错误的（即实质上并非全序的） Ord 实现，也不能产生Ub。（译者注：这一段有点绕，但其中的逻辑并不复杂，即，只要我一直在写Safe代码，就算我写的都是垃圾，就算所有东西都实现错误，那这些Safe代码底层调用的Unsafe代码也不会引发上一篇所提到的那一堆错误中的任何一个——只是程序会表现异常，比如你明明之前向BTreeMap插入了一个值，并且自始至终没有做任何删除，但之后就是找不到这个值）

Unsafe Rust代码就是不能信任Safe Rust代码被正确地编写。BTreeMap 在没有全序的类型上运行会表现异常，不过绝对不会产生UB。Safe的承诺很实。

你可能想问，如果 BTreeMap 不能相信 Ord （就因为它是Safe的）, 为什么它能相信其他 任意 的Safe代码？ 你看， BTreeMap 的实现不也依赖于整数和切片（slice）实现的正确性吗？这些不也都是Safe的吗？

区别在于“范围（scope）”。当 BTreeMap 依赖于整数和切片的时候，它正依赖于非常独特而具体的实现。这个风险是可以估量的，并可与收益进行权衡。在这种情况下，我们反而没什么风险：如果整数或者切片这些在Rust里遍地都是的核心设施实现都有问题，那么 一切 都毁了，还写什么Rust程序啊。而且，维护整数和切片的人也在维护 BTreeMap （都是维护标准库的），因此密切监视以保证这些实现正确还算简单。

同时， BTreeMap 的key类型是泛型参数。如果你信任其 Ord 实现，就相当于你信任过去、现在以及未来的所有 Ord 实现。这风险就大了：有人可能弄错了 Ord 实现，或者干脆撒谎骗过编译器，给他的根本不是全序的类型提供了个所谓 Ord 仅仅是因为“看起来好像能行”。 BTreeMap 得提前为这些情况做好准备。

对于“信任返还给你的闭包（closure）”这种情况同理。

这种没有明确边界的对泛型类型的信任问题也同样是 unsafe traits需要解决的问题，这里就不再赘述了。理论上来说， BTreeMap 类型应该要求它的键（key）实现一个叫 UnsafeOrd 的新trait而不是 Ord ，就像这样：


#![allow(unused)]
fn main() {
use std::cmp::Ordering;

unsafe trait UnsafeOrd {
    fn cmp(&self, other: &Self) -> Ordering;
}
}

于是，一个类型就要用 unsafe 代码来实现 UnsafeOrd 以表明编写的人确信他的实现满足了所有该trait需要的约定。在这种情况下， BTreeMap 底层实现的Unsafe Rust代码就可以有充分理由确信那个 UnsafeOrd 实现是正确的了。如果不正确，那错就在你的trait实现而不在我 BTreeMap ，因为明明是你先掀桌子，违反了Rust的安全保证。

是否要给trait标记成 unsafe 实际上是个API设计决策。Safe trait实现起来简单，但依赖这些trait的unsafe代码必须防止错误行为。将trait标记成 unsafe 把正确实现的重任移交给了实现者。按照惯例，Rust会尽可能避免将trait标记成 unsafe 的，因为这么做会让Unsafe Rust像癌细胞一样充斥在标准库的各个角落，这可不好。

Send 和 Sync 是Unsafe的，因为线程安全（thread safety）是程序设计里的一个 基本问题 ， Unsafe代码完全无法像抵御错误的 Ord 实现一样抵御它。同样， GlobalAllocator 记录了程序所使用的所有堆内存， 像 Box 或 Vec 这样的（泛型）类型都要依赖于它。如果它被搞砸了（比如分配了一块仍然还被程序使用着的内存），我们没法检测到这个问题，并且束手无策。因此 GlobalAllocator 也必须是Unsafe的（译者注：否则问题和上面同理——“我写的全是Safe Rust，怎么会出内存错误呢？”）。

要不要把你写的trait标记成 unsafe 取决于同样的考虑。如果 unsafe 代码不能承受违反你trait约定的实现的后果，那么把你的trait标记成 unsafe 就合情合理。

另外，虽然 Send 和 Sync 是 unsafe traits，如果一个类型内部派生的所有部分也都被证明实现这些traits是安全的，那这个类型本身 也 会被自动实现这些traits。内部值类型都是 Send 的类型也自动是 Send 的（否则不是），内部值类型都是 Sync 的类型也自动是 Sync 的（否则不是）。这种行为降低了标记这两个trait为 unsafe 的各式各样的危险情况。而且，自己 手动实现 内存分配器（或是因故要直接操作内存分配器）的人少之又少，这也降低了 unsafe 的风险。

以上就是Safe Rust和Unsafe Rust取得的平衡。这种分离的设计目的是让使用Safe Rust尽可能符合人体工程学（ergonomic），不过它又要求你在写Unsafe Rust的时候要额外小心。本书的其余部分会广泛讨论你要小心什么，以及Unsafe Rust要遵守什么约定。

Unsafe Rust能做什么

Unsafe Rust只比Safe Rust多了这些：

  • 解引用（dereference）裸指针
  • 调用 unsafe 函数（包括C函数，编译器黑魔法(intrinsics)、裸内存分配器）
  • 实现 unsafe traits
  • 改变 static 变量的值
  • 访问 union 内的数据域

就这么多了。上面这些操作“沦为”Unsafe的原因是，误用它们会导致恐怖的UB。让UB入关就意味着允许编译器做任何错误得操作。你 绝不能 容忍UB。

不过与C不同的是，Rust中能滋生UB的土壤少得可怜。Rust语言核心的全部注意力都集中在防止以下事情：

  • 通过 * 操作解引用悬空指针或未对齐的（unaligned）指针（见下面的讨论）
  • 违反 指针别名规则
  • 以错误的ABI方式调用函数，或从错误的展开ABI（unwind ABI）的函数调用中展开（unwind）
  • 导致 数据竞争 （data race）
  • 执行通过 target features 编译的代码，但当前线程不支持该设置
  • 生成非法值（可能就它自己，也可能是作为像 enum/struct/数组/元组的一个域）：
    • 非0非1的 bool 值
    • 不符合任何枚举标签的枚举值（译者注：一个枚举可以看作“(A, B)”元素的集合，其中A是字符串标签，B是对应的整数值。这里严格来说，非法值形如(_, C)，其中C\( \notin \)BSet，BSet是所有合法B的集合）
    • 空 fn 指针
    • 不在 [0x0, 0xD7FF] 或 [0xE000, 0x10FFFF] 范围内的 char 值
    • !（译者注：即never type，在《The Book》里19.3节有讲到）（任何值对它都是非法的）
    • 从 未初始化内存 读来的整数（i*/u*）、浮点数（f*）、裸指针，或 str 内有未初始化（uninitialized）的内存
    • 悬空、未对齐、指向非法值的引用/Box
    • 所有拥有如下所述的错误元数据的广义引用（wide reference）和 Box：
      • 如果一个 Trait 的指针或引用实质上并不指向 dyn Trait 中运行时（通过虚表，vtable）才能确定的那个真实实质类型，那么 dyn Trait 的元数据是非法的
      • slice的元数据在其length不是一个合法的 usize 时非法（比如这个usize是从未初始化内存里读来的）
    • 如果一个类型明确规定其中的某个（些）值是非法的，但实际上这个类型恰好就存着这个（些）非法值，比如 NonNull 里存着null值的情况。（访问这些非法值的功能目前是unstable的，不过像 NonNull 这样的stable libstd类型就使用了这个功能）

当你进行赋值、传参给函数或底层系统原语、从函数或底层系统原语返回值时，其实都在“生成”（produce）一个值。

如果指针/引用是空的，或者它指向的内存区域空间不都是（或都不是）之前以某种方式被正确分配的，那么它就是悬空指针（译者注：Rust与C/C++对于悬空指针的定义完全一致；这里的描述多少有些学院派）。被指向的内存字节范围（span）的大小取决于指针的值，以及被指向类型的大小。因此，如果这个范围是空的，那么此时“悬空”与“非空”定义等价（译者注：如果被指向的类型大小为0，那这个指针理应是空的，否则它总归指向了不属于他的某个起始地址）。注意切片和字符串拥有它们所指向的范围下的全部空间，它们的长度（length）元数据不能过大。尤其是，它们都不能拥有比 isize::MAX 字节还大的内存。万一你真想要这么大，考虑使用裸指针管理它们吧。

讲完了。这些就是Rust里UB的所有源头了。当然，unsafe函数和unsafe traits可以很自然地声明任意约束来要求程序去遵守它们，以避免UB。比如内存分配器API就明确写了：回收（deallocate）未被分配的（unallocated）内存是UB。

然而，违背这些约束会直接或间接地引发如上所述的问题。编译器内部搞了一些更激进的黑魔法来保证代码可以怎么被优化。比如，编译器对Vec和Box有更激进的优化，因为它知道，这些结构内部的指针必定非空（译者注：可以理解为它们的二进制序列绝对不会有全零这个非法表示）。

不过Rust允许一些奇怪的操作，而且不认为这些操作是 unsafe 的：

  • 死锁
  • 竞争条件
  • 内存泄漏
  • 析构失败
  • 整数溢出
  • 程序中止（abort）
  • 删库跑路

当然了，能妥善处理如上所有问题的程序也 可能 是不正确的。Rust做了很多努力，提供了很多工具让这种事尽可能少发生，但终归无法杜绝。

与Unsafe共舞

Rust仅仅在很有限的范围让你编写Unsafe Rust，不过不要以为这时候你只需要考虑Unsafe部分就够了。考虑以下函数：


#![allow(unused)]
fn main() {
fn index(idx: usize, arr: &[u8]) -> Option<u8> {
    if idx < arr.len() {
        unsafe {
            Some(*arr.get_unchecked(idex))
        }
    } else {
        None
    }
}
}

这个函数即安全又正确。我们确保索引idx不越界，若不越界，我们就通过unsafe语句块获取数组的对应索引值。我们把这种通过Unsafe方式但又正确实现的函数称为 sound 的，意思是safe代码不会因为该函数产生UB（还记得吗，这是Safe Rust最基本的原则）。

但就算这么普通的函数，unsafe块的范围也是个应该讨论的议题。如果把这个函数的 < 换成 <=：


#![allow(unused)]
fn main() {
fn index(idx: usize, arr: &[u8]) -> Option<u8> {
    if idx <= arr.len() {
        unsafe {
            Some(*arr.get_unchecked(idx))
        }
    } else {
        None
    }
}
}

这个程序目前就是 unsound 的了。Safe Rust也产生了UB，仅仅因为 我们更改了Safe块的代码 。这就是安全的基本问题：安全并不是局部的。Unsafe操作的soundness必须依赖于某些已经建立好的“Safe”操作状态。

从某个角度来讲，安全是模块化的，因为切换到不安全模式并不要求你去费尽心思思考Unsafe所能引发的所有坏结果。举例来说，不检查越界地访问slice这件事绝对不需要你担心slice是不是null的，或者slice内包含了未初始化的内存——其他基本方面都没有改变。不过从另一个角度来讲，安全又 不是 模块化的，因为程序在内部是有状态的，你的Unsafe操作可能依赖于任意其他状态。

当你不经意间打破一种持久的状态时，这种非模块化的糟糕之处就体现出来了。考虑一种 Vec 的简单实现：

use std::ptr;

// 注意：严格来讲下面的定义过于简单（且有问题）。见后面“实现Vec”的章节。
// 译者注：先不论下面提到的ZST（零size类型）问题。C/C++等语言可以这么实现，但Rust不行，因为
// 这个结构没有表达出“Vec实际上存储并拥有一系列T值”的语义，在更复杂的调用中会引发生命周期问题
// 而导致无法编译；至少在Vec中要加上`std::marker::PhantomData`标记。当然这还不是全部。读者
// 若有兴趣，可以看一下标准库Vec的实现，其中涉及多层嵌套结构（借助零开销抽象）、永恒非空指针、
// ZST等技术。
pub struct Vec<T> {
    ptr: *mut T,
    len: usize,
    cap: usize,
}

// 注意这个实现没有正确处理ZST。见后面“实现Vec”的章节。
impl<T> Vec<T> {
    pub fn push(&mut self, elem: T) {
        if self.len == self.cap {
            // 尚未实现
            self.reallocate();
        }
        unsafe {
            ptr::write(self.ptr.add(len), elem);
            self.len += 1;
        }
    }
    fn reallocate(&mut self) { }
}

fn main() {}

这些代码足够简单，想要形式化地验证它是正确的不算太难。现在考虑添加如下方法：

fn make_room(&mut self) {
    // 增长Vec容量
    self.cap += 1;
}

这段代码是100%的Safe Rust，但也是100%的unsound。改变capacity违反了Vec的不变式（即， cap 反映的是Vec通过请求分配所得到的空间范围）。Vec本身无法确认它是否还能通过自增 cap 擅自吞下更多空间。Vec 必须 无条件信任capacity域，因为我们没法通过别的方式验证它。

因为它依赖于struct域提供的不变式， unsafe 代码不仅仅污染了整个函数，它还污染了 整个模块 。通常，限制Unsafe代码范围的唯一方式在于模块边界与私有性（privacy）。

然而目前为止它们工作得 很好。 make_room 的存在 不 影响Vec的soundness，因为我们没有让它变成 pub 的。目前，只有定义了这个函数的模块才能调用它。另外， make_room 直接访问了Vec的私有域，所以它也只能被定义在与Vec相同的模块里。

所以，我们可以基于一系列复杂的不变式来建立Safe的抽象。这对于维护Safe Rust和Unsafe Rust之间的关系 至关重要 。

我们已经看到了，Unsafe代码必须信任 一部分 Safe代码，但不该信任 泛型 Safe代码（译者注：回忆之前介绍的 Ord 与 BTreeMap ）。私有性对于Unsafe代码也很重要，原因类似，因为私有性可以让我们没有必要担心“任何Safe代码都可能搞砸目前的稳定不变式”这种不可能的事情。

一路顺风，注意安全！
